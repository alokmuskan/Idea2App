────────────────────────────────────────────────────────────────────
  ARCHITECTURE FLOW
────────────────────────────────────────────────────────────────────

1. PROJECT BRIEF (User)
   - 4-question guided intake before any code is written
   - What problem · Who it's for · Core action · Inspiration
   - AI synthesizes a Product Brief, user confirms
   - POST /api/chat with full message history

2. NEXT.JS API ROUTE
   - Keeps all LLM API keys server-side
   - Injects system prompt
   - Routes to chosen provider

3. INTENT CLASSIFIER (GPT-4o-mini)
   - Classifies every message before sending to main model
   - create  → full generation mode
   - refine  → diff-only mode (only changed files)
   - debug   → error fix mode
   - explain → describe code, no file output

4. PLANNING STEP (same fast model)
   - Breaks complex requests into ordered task list
   - Output wrapped in <plan> tags
   - Rendered as visible Blueprint card in UI before generation
   - One cheap API call, makes system feel agentic

5. MAIN AI GENERATOR (GPT-4o / Claude)
   - Receives: system prompt + project state JSON + message history
   - Outputs: <plan> tasks + <project> files in one stream
   - On refinement: outputs only changed files

6. STREAM PARSER
   - Reads SSE stream chunk by chunk
   - Extracts <file> blocks → WebContainer FS (parallel writes)
   - Extracts <shell> blocks → command queue (sequential)
   - Updates Project State JSON as files are written
   - Counts files written per message (for "X files changed" badge)

7. WEBCONTAINER
   - Full Node.js in browser via WASM
   - Receives files in parallel
   - Runs commands sequentially: npm install → vite --host
   - Error detected → capture stack trace → send to AI → patch → reload

8. LIVE PREVIEW
   - Split pane: chat left, preview right (always both visible)
   - iframe showing running app on :5173
   - Route map sidebar (all pages listed, clickable)
   - AI-generated app name + SVG logo injected into app

9. DEPLOY + GITHUB
   - /api/deploy → Vercel API → live URL
   - /api/github → GitHub OAuth → create repo → commit all files
   - "Push & Deploy" button fires both simultaneously
   - Both URLs saved to Supabase deployments table

10. SUPABASE (background writes, non-blocking)
    - Every prompt saved to prompts table
    - File tree saved to snapshots table on each deploy
    - Vercel URL + GitHub URL saved to deployments table

────────────────────────────────────────────────────────────────────
  THE HARDEST PART OF THE ENTIRE SYSTEM
────────────────────────────────────────────────────────────────────

  The hardest part is NOT the AI.

  It is: Stream Parser + WebContainer File Writes

  You must do ALL of this correctly and in order:
    1. Read streaming LLM response chunk by chunk
    2. Detect <file> blocks as they arrive mid-stream
    3. Extract the file path from the tag
    4. Write file contents to WebContainer filesystem
    5. Queue <shell> commands separately
    6. Execute shell commands ONLY after ALL files are written

  The correct order is non-negotiable:
    write all files → confirm writes done → run shell commands

  If done wrong:
    npm install runs before files exist → everything fails silently

  Common mistakes:
    - Running npm install while files are still streaming
    - Not waiting for WebContainer FS write promises to resolve
    - Parsing file content incorrectly due to chunked stream boundaries
      (a chunk boundary can split mid-tag — your parser must buffer)


────────────────────────────────────────────────────────────────────
  WHY WEBCONTAINERS FEEL HARD AT FIRST
────────────────────────────────────────────────────────────────────

  WebContainers run Node.js inside the browser via WASM.
  Things behave slightly differently from normal Node:

    Filesystem API    → async, promise-based, not fs.writeFileSync
    Ports             → must use --host flag or iframe can't reach :5173
    Install time      → npm install is slower than local, be patient
    Memory limits     → browser tab has memory ceiling, keep apps lean
    COOP/COEP headers → your Next.js host must set these headers or
                        WebContainers won't initialise at all

  ⚠️  THIS IS THE #1 SILENT FAILURE — ADD THIS BEFORE YOU START  ⚠️

  Without these headers WebContainers throw a cryptic error and
  nothing works. It looks like a code bug. It is not. It takes
  2 minutes to fix if you know about it. Hours if you don't.

  Add this to next.config.js BEFORE writing any other code:

    const nextConfig = {
      async headers() {
        return [
          {
            source: "/(.*)",
            headers: [
              {
                key: "Cross-Origin-Embedder-Policy",
                value: "require-corp",
              },
              {
                key: "Cross-Origin-Opener-Policy",
                value: "same-origin",
              },
            ],
          },
        ];
      },
    };
    module.exports = nextConfig;

  Do this on line 1 of your project. Not hour 10.

  Most WebContainer bugs are environment bugs, not code bugs.
  When something breaks, check these before touching your logic.

  CORRECT BUILD ORDER (FOLLOW THIS EXACTLY)
────────────────────────────────────────────────────────────────────

  STEP 1 — WebContainer smoke test (3 hours)
  ───────────────────────────────────────────
  Before writing any AI code, manually prove WebContainers work.

  Hardcode these two files and write them to WebContainer FS:
    - package.json  (minimal React + Vite)
    - src/main.jsx  (just renders <h1>Hello</h1>)

  Then run sequentially:
    npm install
    npm run dev

  Open the iframe pointing at :5173.
  If you see "Hello" → WebContainers work. Continue.
  If you don't → fix this before touching anything else.
  Do NOT skip this step. Everything depends on it working.

  STEP 2 — AI generates files, you write them (4 hours)
  ───────────────────────────────────────────────────────
  Send this exact prompt to your AI:
    "Generate a minimal React Vite project with Tailwind CSS.
     Output files using <file path="..."> tags."

  Take the AI response (non-streaming first, simpler to debug).
  Parse it manually.
  Write those files into WebContainer.
  Run the server.
  Confirm preview works.

  STEP 3 — Build the stream parser (6 hours)
  ────────────────────────────────────────────
  This is the hardest step. Take the most time here.

  Switch from non-streaming to streaming response.
  Parse <file path="..."> blocks as chunks arrive.
  Key challenge: chunk boundaries can split mid-tag.
  You MUST buffer incomplete tags across chunks.

  Parser logic:
    - Maintain a buffer string
    - Append each chunk to buffer
    - Check if buffer contains complete <file>...</file> block
    - If complete → extract path + content → write to FS → clear
    - If incomplete → keep buffering → wait for next chunk
    - Collect <shell> commands separately into a queue
    - After stream ends → execute shell queue sequentially

  Test with slow streaming (add artificial delay) to catch
  edge cases where tags split across chunk boundaries.

  STEP 4 — Add planning step (2 hours)
  ──────────────────────────────────────
  Parse <plan> tags from the stream.
  Render them as the Blueprint card UI before files appear.
  Show tasks ticking off as corresponding files are written.

  STEP 5 — Add refinement mode (3 hours)
  ────────────────────────────────────────
  Wire up intent classifier.
  On refine → tell AI to output only changed files.
  Count files written → show "X files changed" badge.
  Confirm hot-reload works without full server restart.

  STEP 6 — Add deploy + GitHub (4 hours)
  ────────────────────────────────────────
  /api/deploy → Vercel API
  /api/github → GitHub REST API
  "Push & Deploy" combined button
  Save URLs to Supabase
